# AURA OS — Динамический харнес

> **Модель — commodity, harness — moat.** Одна и та же модель в разной обвязке даёт кратную разницу качества. AURA OS — это harness-оркестратор: она строит обвязку под конкретную задачу, а состояние держит в файлах и переменных оркестратора, не в контексте модели (как в Claude Code Dynamic Workflows).

## Возможности харнеса

| Возможность | Что делает |
|---|---|
| **Dynamic Harness** — авто-выбор паттерна под задачу | single · loop-until-done · fan-out-synthesize · adversarial-verification · tournament · generate-and-filter · classify-and-act (7 паттернов) |
| **Ralph Loop** — агент в цикле до готовности | автономия L1 (только отчёт) · L2 (с подтверждением) · L3 (автономно), лимит итераций, backpressure, cost-guard (жёсткий лимит $) |
| **CONSTRAINTS.md** — правила в каждый прогон | ручное добавление + автозагрузка + Self-Improving Loop (правила копятся из вердиктов верификатора) |
| **Smart Model Routing** | модель под сложность подзадачи (trivial→Haiku, standard→Sonnet, complex→Opus) + Smart Zone |

## 1. Dynamic Harness — паттерны

AURA сама классифицирует задачу и выбирает паттерн оркестрации (вкладка **Харнес → Авто-харнес**):

- **single** — узкая задача, один проход агента.
- **loop-until-done** — проверяемая цель с итерациями (Ralph loop).
- **fan-out-synthesize** — широкая задача по многим объектам: разбить на ветки, выполнить параллельно, синтезировать.
- **adversarial-verification** — один делает, другой ищет дыры (прокурор/адвокат), затем фикс.
- **tournament** — несколько подходов решают одно, судья выбирает лучший.
- **generate-and-filter** — сгенерировать много кандидатов → отфильтровать по критериям → топ-3.
- **classify-and-act** — классифицировать поток входа по типам → обработать каждый своим способом.

«Показать план» выводит выбранный паттерн и обоснование без запуска.

## 2. Ralph Loop

Агент идёт к цели маленькими шагами, перезапускаясь, пока не достигнет (или до лимита итераций). Завершение — по маркеру `AURA_LOOP_DONE` или файлу `done.txt` в рабочей папке.

- **Backpressure** — тест-команда (напр. `npm test`), которая прогоняется после каждой итерации как ограничитель; результат подаётся в следующую итерацию.
- **Автономия:**
  - **L3** — автономно. Идёт сам до готовности/лимита.
  - **L2** — пауза с подтверждением между итерациями.
  - **L1** — только отчёт: агент НЕ меняет проект, а предлагает план.
- **Cost-guard** — оценка расхода до запуска и жёсткий лимит в долларах (защита от «$4000 за ночь — забыли, что loop работает»).

> Принцип фазового внедрения: новый цикл начинать с L1 → L2 → L3, не прыгая через уровни.

## 3. CONSTRAINTS.md — Self-Improving Loop

`<vault>/AURA/CONSTRAINTS.md` автозагружается в начало каждого прогона. Правила добавляются вручную (вкладка **Харнес → Правила**) или **сами копятся** из вердиктов верификатора: каждая найденная ошибка → правило → следующий прогон её не повторит. Переобучается система вокруг модели, а не модель.

## 4. Smart Model Routing + Smart Zone

Планировщик помечает подзадаче `complexity`, оркестратор выбирает модель:
`trivial → Haiku · standard → Sonnet · complex → Opus` (+ профили для Codex/Gemini/Kimi).

**Smart Zone** (Dex Horthy, 100K+ сессий): <40% окна — модель максимально умна; 40–60% — деградирует; >60% — «уплотняет» историю и считает невыполненное выполненным. AURA следит за заполнением и советует дробить на свежие контексты.

## Принцип CLI > MCP

Для большинства задач bash-команды экономнее MCP по токенам: MCP сливает описания всех тулов в контекст на старте каждого сообщения, bash — ноль накладных. AURA поиск по базе знаний работает через `rg` (CLI-first). MCP — для типизированных интеграций, OAuth, чувствительных данных.
