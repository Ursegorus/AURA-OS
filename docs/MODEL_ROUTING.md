# AURA OS — Smart Model Routing (умная маршрутизация моделей)

> Авто-распределение подзадач по моделям разной мощности для экономии токенов/лимитов
> без потери качества на сложных шагах. Опциональная фича оркестратора.

## Зачем
Внутри одной задачи AURA подзадачи сильно различаются по сложности: «переформатируй файл»
дешевле гнать на маленькой модели, а «спроектируй модуль» требует флагмана. Цены различаются
кратно (для Claude: Haiku ~5× дешевле Opus, Sonnet ~1.67× дешевле). Если координатор на этапе
планирования сам проставит сложность подзадачи, оркестратор подберёт модель — и стоимость задачи
падает в разы там, где не нужен флагман.

## Идея
1. На этапе **PLAN** координатор для каждой подзадачи возвращает поле `complexity`:
   `trivial | standard | complex`.
2. На этапе **EXECUTE** оркестратор по карте `agent.models[complexity]` выбирает модель и
   передаёт её агенту; раннер подставляет её CLI-флагом (`agent.modelFlag`, напр. `--model`).
3. Фича включается настройкой **`smartRouting`** (по умолчанию выкл). Выкл → агент работает
   на своей дефолтной модели, поведение не меняется.

### Карта сложности → модель (встроенный профиль Claude Code)
| complexity | Модель (alias) | Когда |
|---|---|---|
| `trivial` | `haiku` | форматирование, конвертация, простые правки, переименования |
| `standard` | `sonnet` | обычный код, ресёрч, тексты, ревью |
| `complex` | `opus` | архитектура, сложный кодинг, отладка, многошаговое рассуждение |

Профиль задаётся на агенте: `modelFlag: '--model'`, `models: { trivial:'haiku', standard:'sonnet', complex:'opus' }`.
Для агентов без профиля (Codex/Gemini/Ollama) роутинг не применяется — работают на своей модели.
Свой агент может задать `modelFlag` + `models` и тоже получить тиринг.

## Контракт планировщика
В JSON-плане у подзадачи добавляется поле:
```json
{ "id":"t1", "title":"...", "agent":"claude-code", "role":"coder",
  "complexity":"complex", "prompt":"...", "dependsOn":[] }
```
Если поле отсутствует/невалидно → `standard` (безопасный дефолт).

## Поток исполнения
```
PLAN → у каждой подзадачи complexity
  └─ EXECUTE: model = smartRouting ? agent.models[complexity] : undefined
       └─ agents.run(agent, prompt, { model })
            └─ если model и agent.modelFlag → args = [modelFlag, model, ...args]
REVIEW/FIX: исправление наследует complexity целевой подзадачи; повторное ревью — standard
```

## Безопасность поведения / совместимость
- Выкл по умолчанию → нулевой риск для существующих сборок.
- Модель инжектится ТОЛЬКО если у агента есть `modelFlag` и `models[tier]` — иначе игнор.
- Никакого авто-переключения «на лету» вне плана: модель фиксируется на старте подзадачи.

## Что это даёт продукту
- Прямая экономия токенов на смешанных задачах (часто 2–4×) без ручного выбора модели.
- Сильный аргумент Pro-тарифа: «умная экономия» + лог выбранной модели по подзадаче.
- Ложится в H2/H3 роадмапа.
